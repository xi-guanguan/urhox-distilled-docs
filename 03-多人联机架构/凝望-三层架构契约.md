# 三层架构契约（server / client / shared）

> **来源**: 社区分享（微信群）
> **作者**: 凝望
> **蒸馏等级**: L2（学习参考）
> **适用场景**: 多人游戏项目的代码分层架构，常驻服务器（Dedicated Server）模式
> **核心价值**: 三层目录架构 + Schema 驱动注册 + Handler-Service-PDM 服务端分层 + Dirty-Cache 持久化
> **版本**: V2（2026 更新版，新增服务端分层、Schema 注册机制、持久化模型）

## 一、目录结构

```
scripts/
├── shared/          ← 零依赖（纯定义：Schema/Consts/Defs/纯函数）
│   ├── schemas/
│   │   └── CharacterSchema.lua  ← 中心 Schema（全项目唯一）
│   └── xxx/
│       ├── XxxSchema.lua        ← 子系统 Schema（注册到 CharacterSchema）
│       └── XxxConsts.lua
├── server/          ← 可 require shared，禁 require client
│   ├── character/
│   │   └── PlayerDataManager.lua  ← PDM（全项目唯一）
│   └── xxx/
│       ├── XxxHandler.lua
│       └── XxxService.lua
└── client/          ← 可 require shared，禁 require server
    ├── data/
    │   └── PlayerStore.lua        ← 数据访问层（全项目唯一）
    └── xxx/
        └── XxxClient.lua
```

## 二、依赖方向（不可逆）

```
shared ← server ← [网络事件] → client → shared
```

| 来源 | 可 require | 禁止 require |
|------|-----------|-------------|
| shared/* | 无（零依赖） | server/*, client/* |
| server/* | shared/* | client/* |
| client/* | shared/* | server/* |

**判定规则**: 如果一个模块删掉引擎后还能 `lua` 命令加载，才能放 shared/。

### shared/ 允许 vs 禁止

| 允许 | 禁止 |
|------|------|
| Schema 字段定义、事件名常量 | 任何业务逻辑 |
| 枚举/类型定义、游戏数据表 | serverCloud/PDM 读写 |
| 纯函数工具（无引擎依赖） | UI 组件、引擎 API、事件订阅 |

## 三、Schema 驱动注册机制

三个中心化单例通过 CharacterSchema 联动：

```
XxxSchema.Fields
      │
      │ RegisterSubsystemFields
      ▼
CharacterSchema.Fields ──────┬────────────────────┐
                             │                    │
                             ▼                    ▼
                  PDM 动态构建键注册表    PlayerStore 动态构建 keyMap
                (SCALAR/STRING/JSON_KEYS)   (fieldKey → pdmKey)
```

### 子系统 Schema 模板

```lua
-- shared/xxx/XxxSchema.lua
local XxxSchema = {}

XxxSchema.Fields = {
    xxxLevel = {
        pdmKey  = "XxxLevel",
        type    = "scalar",       -- scalar | string | json
        persist = { via = "cloud", cloudKey = "xxx_level" },
        desc    = "XXX 等级",
    },
    xxxData = {
        pdmKey  = "XxxData",
        type    = "json",
        persist = { via = "cloud", cloudKey = "xxx_data" },
        desc    = "XXX 数据（JSON）",
    },
    xxxConfig = {
        pdmKey  = "XxxConfig",
        type    = "string",
        persist = false,          -- 不持久化（仅运行时）
        desc    = "XXX 临时配置",
    },
}

return XxxSchema
```

### 注册到 CharacterSchema

```lua
-- shared/schemas/CharacterSchema.lua 底部
local XxxSchema = require("shared.xxx.XxxSchema")
CharacterSchema.RegisterSubsystemFields("Xxx", XxxSchema.Fields)
-- PDM 和 PlayerStore 自动包含新字段，无需修改
```

**新子系统接入只需 2 步**：写 Schema + 注册。

## 四、服务端三层分离（Handler → Service → PDM）

```
客户端请求
    │
    ▼
Handler（网络入口）
    │ 参数校验、调 Service
    ▼
Service（业务逻辑）
    │ 纯逻辑，调 PDM 读写
    ▼
PDM.SetStat / GetStat（数据操作）
    │ SetVar(REPLICATED) → 客户端 PlayerStore 自动更新
    │ MarkDirty → 心跳 BatchCommit → serverCloud
    ▼
客户端 PlayerStore.Get()
```

### Handler 标准模式

```lua
function XxxHandler.OnAction(uid, data)
    -- 1. 参数校验
    if not data.param then return SendFail(uid, EVENT, "missing param") end

    -- 2. 调 Service
    local ok, err = XxxService.Action(uid, data.param)
    if not ok then return SendFail(uid, EVENT, err) end

    -- 3. 先同步再发结果（MarkDirty 有 0.1s 延迟）
    PDM.SyncXxx(uid)
    ServerSync.SendToClient(uid, EVENT_RESULT, { ok = true })
end
```

**原则**：
- Handler 不做逻辑，只做路由和校验
- Service 不做网络 IO，只做计算和调 PDM
- PDM 不做业务判断，只做数据读写

### Service 标准模式

```lua
function XxxService.Action(uid, param)
    local current = PDM.GetStat(uid, "xxxLevel")
    if current >= MAX_LEVEL then
        return false, "已满级"
    end
    PDM.SetStat(uid, "xxxLevel", current + 1)
    return true
end
```

## 五、持久化模型（Dirty-Cache + BatchCommit）

```
SetStat / SetStruct
    │
    ├── 更新内存
    ├── SetVar(REPLICATED) → 客户端实时同步
    └── MarkDirty          → 标记待持久化
                                │
                         心跳信号（定时）
                                │
                                ▼
                          BatchCommit
                                │
                        收集所有脏数据
                                │
                                ▼
                       serverCloud 批量写入
```

### 断线处理顺序（铁律）

```lua
-- ✅ 正确：先保存再清理
function OnPlayerDisconnect(uid)
    PDM.SaveAll(uid)        -- 先保存
    PDM.UnloadPlayer(uid)   -- 再清理内存
end

-- ❌ 错误：先清理再保存 = 数据回档
function OnPlayerDisconnect(uid)
    PDM.UnloadPlayer(uid)   -- 内存已清 → SaveAll 无数据可存
    PDM.SaveAll(uid)
end
```

## 六、客户端数据访问

客户端通过 PlayerStore 读取服务端同步的数据，**只读不写**：

```lua
-- client/data/PlayerStore.lua 自动从 REPLICATED 节点同步
local level = PlayerStore.Get("xxxLevel")

-- 修改数据必须发请求到服务端
SendToServer(EVENT_ACTION, { param = value })
```

**禁止**：客户端缓存一份数据自行修改 → 会与服务端不一致。

## 七、新建子系统流程

1. **shared**: 创建 `XxxSchema.lua`（定义字段）+ `XxxConsts.lua`（事件名/枚举）
2. **shared**: 在 `CharacterSchema.lua` 底部 require + RegisterSubsystemFields
3. **server**: 创建 `XxxHandler.lua`（网络入口）+ `XxxService.lua`（业务逻辑）
4. **client**: 创建 `XxxClient.lua`（事件总线 + SendToServer）
5. 通过 `PDM.GetStat/SetStat` 读写，通过 `PlayerStore.Get` 客户端访问

---

## 八、九条铁律

1. **依赖单向**：shared←server, shared←client, 跨层只走网络事件
2. **shared 零副作用**：禁引擎 API、禁全局状态修改
3. **PDM 单一数据源**：禁客户端缓存、禁绕过 PDM 的 SendToClient
4. **Handler 不做逻辑**：逻辑全在 Service
5. **Sync 在 Send 前**：MarkDirty 有 0.1s 延迟，发结果前必须 SyncXxx
6. **断线先存后清**：SaveAll → UnloadPlayer（反了 = 回档）
7. **cjson 是全局**：`local cjson = cjson`，禁止 require
8. **背包双写**：table.remove + ListDelete 缺一不可
9. **模块自建事件总线**：禁依赖旧版 Client.Emit

---

## 九、迁移规则

旧文件替换为转发桩：`return require("shared.events.NetworkEvents")`。新代码必须用新路径。
