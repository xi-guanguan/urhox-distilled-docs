# 常驻服务器开发规范

> **这是什么**: UrhoX 常驻服务器（Dedicated Server）开发的强制架构规范，定义三层目录架构、服务端三层分离（Handler→Service→PDM）、Schema 驱动数据定义、GateManager 门控、操作队列串行化等核心规则。
> **原作者**: 凝望
> **推荐度**: L2（参考学习）— 完整的服务器架构约束体系，适合中大型多人游戏项目。
> **适用场景**: 使用常驻服务器（persistent_world/server_authoritative）开发多人游戏时。

---

## 1. 三层目录架构

```
scripts/
├── shared/    ← 零依赖（纯定义，不 require 任何非 shared 模块）
├── server/    ← 可 require shared/，禁止 require client/
└── client/    ← 可 require shared/，禁止 require server/
```

### shared/ 内容约束

| 允许 | 禁止 |
|------|------|
| Schema 字段定义、事件名常量 | 任何业务逻辑（if/else） |
| 枚举/类型定义、静态数据表 | serverCloud 读写、PDM/SetVar |
| 纯函数工具（无引擎依赖） | UI 创建、事件订阅、有副作用函数 |

**判定法**: 删掉引擎后还能 `lua` 命令加载成功 → 可放 shared/。

---

## 2. 服务端三层分离：Handler → Service → PDM

| 层级 | 职责 | 禁止 |
|------|------|------|
| **Handler** | 事件注册、参数解析、**业务规则校验**、调用 Service | 直接调用 PDM/serverCloud |
| **Service** | 封装 PDM 读写，提供 Get/Save/Find/Add 纯数据 API | 业务逻辑、事件注册、网络发送 |
| **PDM** | 持久化存储 + 客户端自动同步 | — |

调用方向（不可逆）：`Handler → Service → PDM`

### 关键规则

- Handler 通过 Service 间接操作 PDM，**禁止直接调用 PDM**
- Service 只包含纯数据操作（Get/Save/Find/Add），不含 if/else 业务校验
- Handler 和 Service **必须是独立文件**，不合并
- Handler 每个执行路径末尾**必须调用 `done()`**，否则操作队列永久阻塞

### Service 模板

```lua
local PDM = require("server.character.PlayerDataManager")
local XxxService = {}
function XxxService.GetAll(uid) return PDM.GetStruct(uid, "XxxData") or {} end
function XxxService.SaveAll(uid, data) PDM.SetStruct(uid, "XxxData", data) end
return XxxService
```

### Handler 模板

```lua
function XxxHandler.Init(pdm, server)
    Server_ = server
    server.On(NetworkEvents.REQUEST_XXX, function(uid, eventData, done)
        local data = XxxService.GetAll(uid)   -- 通过 Service 读
        -- 业务校验...
        XxxService.SaveAll(uid, data)          -- 通过 Service 写
        Server_.SendToClient(uid, EVT, { Ok = true })
        done()  -- 必须调用！
    end)
end
```

## 3. Schema 驱动数据定义

每个子系统在 `shared/schemas/` 创建独立 Schema：

```lua
XxxSchema.Fields = {
    fieldKey = {
        pdmKey   = "PdmKeyName",
        type     = "scalar"|"string"|"json",
        persist  = { via = "cloud", cloudKey = "xxx" } | false,
        desc     = "字段说明",
    },
}
```

### 持久化策略

| persist | 含义 |
|---------|------|
| `{ via = "cloud", cloudKey = "xxx" }` | serverCloud scores 自动读写 |
| `{ via = "money", cloudKey = "xxx" }` | 委托 ServerMoney provider |
| `{ via = "list", cloudKey = "xxx" }` | 委托 ServerList provider |
| `false` | 仅 SetVar 运行时同步，不写入 serverCloud |

**唯一性**: 一个字段只能出现在一个 Schema 中。消费方从 Schema 读取，禁止硬编码重复定义。

## 4. GateManager 门控

为每个玩家维护命名门控锁，防止同一玩家同一资源的并发异步操作。

| 决策 | 选择 |
|------|------|
| 纯同步操作 | `Server.On` — 操作队列已保证串行 |
| 含异步回调（serverCloud/money） | `Server.OnGated` — 防回调期间同类请求并发 |

```lua
server.OnGated(NetworkEvents.REQUEST_XXX, "gateName",
    NetworkEvents.XXX_RESULT, HandleXxx)
```

---

## 5. 操作队列

- 每个玩家一个操作队列（OpQueue），handler 自动入队，同一玩家的请求按顺序串行
- handler 签名：`function(uid, eventData, done)`
- **每个执行路径末尾必须调用 `done()`**（含错误分支和异步回调的 ok/error）
- 忘记 `done()` → 该玩家操作队列**永久阻塞**

---

## 6. 数据同步机制

```
服务端 PDM.SetStat/SetStruct → node:SetVar(REPLICATED) → 客户端 SyncClient/PlayerStore
```

### 新增字段分类决策

```
需要跨会话保留？
├─ 是 → 需要实时同步到客户端？
│   ├─ 是 → persist = { via = "cloud" }（持久+实时）
│   └─ 否 → serverCloud 独立读写（仅持久）
└─ 否 → persist = false（仅实时）
```

客户端通过 `PlayerStore.Get("fieldKey")` 统一读取，自动从 Schema 构建 keyMap。

## 7. NetworkEvents 命名规范

| 模式 | 方向 | 示例 |
|------|------|------|
| `REQUEST_` 前缀 | C→S | `REQUEST_EQUIP_SKILL` |
| `_RESULT` 后缀 | S→C | `EQUIP_SKILL_RESULT` |
| `_SYNC` 后缀 | S→C | `CHARACTER_STATE_SYNC` |

所有事件名集中在 `shared/events/NetworkEvents.lua`。

## 8. 新增子系统检查清单

**定义层（shared/）**:
- [ ] `shared/schemas/XxxSchema.lua` 定义 Fields
- [ ] `CharacterSchema.RegisterSubsystemFields` 注册
- [ ] `NetworkEvents.lua` 添加事件常量

**服务层（server/）**:
- [ ] `XxxService.lua` — 纯数据 API
- [ ] `XxxHandler.lua` — 事件注册 + 业务校验
- [ ] 异步操作使用 GateManager
- [ ] 所有路径调用 `done()`

**客户端层（client/）**:
- [ ] `PlayerStore.Get()` 读取（自动可用）
- [ ] `SendEvent` 发请求，监听 `_RESULT`

**质量自查**:
- [ ] Handler 中无 `PDM.*` 直接调用
- [ ] Service 无业务校验/事件注册/网络发送
- [ ] Handler 和 Service 分为独立文件
- [ ] shared/ 模块无副作用
- [ ] require 路径符合依赖方向

## PDM 核心 API 速查

| API | 用途 |
|-----|------|
| `SetStat/GetStat/AddStat` | 标量读写+即时同步 |
| `SetStruct/GetStruct` | JSON 结构读写+即时同步 |
| `AddMoney/RemoveMoney/GetMoney` | 货币操作（委托 provider） |
| `IsLoaded(uid)` | 玩家数据是否已加载 |

**生命周期**: 连接 → LoadPlayer → SetVar 初始化 → 业务处理 → Tick 每 30s 自动存盘 → 断线 → SavePlayer → Dispose

## 禁止模式汇总

| 禁止 | 正确做法 |
|------|---------|
| Handler 直接调用 PDM | 通过 Service 封装 |
| Service 包含业务校验 | 校验放 Handler |
| Service 包含事件注册/网络发送 | 放 Handler |
| Handler+Service 合并单文件 | 各自独立 |
| shared/ 调用引擎 API / 事件订阅 | 放 server/client |
| 消费方硬编码字段定义 | 从 Schema 读取 |
| client require server | 通过 shared 定义+事件通信 |
| 忘记调用 done() | 所有路径都 done() |
